# v0.3.0 Research Summary

## Ellidiss Feature Gap Analysis

### TOP 5 gaps to close:

1. **Code generation** (Ada/C) — both Ellidiss and OSATE generate deployable code. Spar has none. Even skeleton generation would be huge.
2. **Real-time scheduling simulation** — Ellidiss has MARZHIN, OSATE has Cheddar. Spar has static RMA only. Add RTA with timeline visualization.
3. **AADL v2.3 support** — both competitors support AS-5506D. Spar targets v2.2 (AS-5506C).
4. **Formal property checking** — OSATE has AGREE/Resolute, Ellidiss has LMP/LAMP. Spar has none. Lightweight constraint language over HIR.
5. **Security analysis** — Ellidiss has a security rules checker. Neither OSATE nor spar does.

### Spar's existing differentiators (keep/amplify):
- WASM deployment (unique)
- WIT/WAC/wRPC transforms (unique)
- Incremental computation (salsa — unique)
- Fast CLI + JSON (CI-native)
- Lossless CST (refactoring-ready)
- VS Code LSP (10 features — better than Ellidiss's syntax-only extension)

## Architecture Refactoring Workflow

### Proposed `spar migrate` command:

**Input:** Current model + target binding overrides (or extends model)
**Process:**
1. Phase 1: Structural validation (binding targets, connection topology, classifier match)
2. Phase 2: Timing/resource analysis (RMA per processor, E2E latency, bus bandwidth, memory)
3. Phase 3: Safety (EMV2 fault trees, mode analysis, ARINC 653 isolation)
4. Phase 4: Optional formal verification (Lean4 proofs, TLA+ model checking)

**Output:** Verified model + migration report (utilization, latency, bandwidth per component)

### What spar already has:
All of Phase 1-3 analyses exist as the 21 analysis passes. What's new:
- **Diff-based refactoring driver** — takes current + target, applies binding changes, runs pipeline
- **AADL `extends` integration** — target model extends current with overridden bindings
- **Migration report format** — before/after comparison of all metrics

### AADL deployment properties to support:
- `Actual_Processor_Binding`, `Allowed_Processor_Binding`
- `Actual_Memory_Binding`, `Actual_Connection_Binding`
- Bus `Bandwidth`, `Transmission_Time`
- ARINC 653 partition properties

### WASI/WIT to AADL mapping:
- `async func` → Thread (aperiodic/sporadic)
- `stream<T>` → Event Data Port
- `future<T>` → Event Data Port (single-shot)
- WIT `interface` → Feature Group
- WIT `world` → System Type
- Component → Process or System

## v0.3.0 Scope Recommendation

Focus on **architecture refactoring** (tracks 2+3 from planning):
1. `spar migrate --from current.aadl --to target.aadl` command
2. Diff-based binding change analysis
3. Migration report (before/after metrics)
4. Formal verification hooks (Lean4 for scheduling proofs)

Defer to v0.4.0:
- Code generation (large scope)
- AADL v2.3 (parser changes)
- Security analysis
- Scheduling simulation (MARZHIN-style)
